# agent-chrome-wrapper

PyInstaller-frozen Python MCP server, shipped as a self-contained Windows binary (`bin/chrome-wrapper.exe`). End users need no Python toolchain. The MCP starts a separate Chrome instance per session, and is intended to pair with `agent-vdesktop` (launch Chrome here, position/manage the window there) on a shared Windows host.

## Architecture — design decisions (locked; see issues #1–#5 for the build-out)

These are the load-bearing choices the rest of the code assumes. Changing one of them is a re-architecture, not a tweak.

- **One process = one session = one Chrome instance. No multiplexing.** Claude Code spawns exactly one stdio MCP server process per session, so the wrapper *is* the session — it owns a single Chrome instance and never has to route between sessions. Chrome is launched **lazily** on the first tool that needs the browser, not at startup. This is why per-session isolation is essentially free: a separate `--remote-debugging-port` + separate `--user-data-dir` per process, no shared registry of sessions inside one server.
- **stdio is what kills the WSL↔Windows port problem — keep it that way.** The pain this plugin exists to solve: Chrome runs on the Windows host, Claude Code often runs in WSL, and forwarding the CDP debugging port across the WSL boundary was fragile. The fix is that Agent↔wrapper is **stdio only** (no port), and the wrapper drives Chrome over CDP on **Windows-localhost** (`127.0.0.1:<port>`). The debugging port never crosses the WSL boundary. Like `agent-vdesktop`, the `.exe` runs as a *Windows* process even when invoked from WSL (binfmt_misc). Do not "expose" the CDP port or add a network transport — that re-introduces the exact problem the design removes.
- **Driver is Raw CDP (WebSocket), not Selenium/Playwright.** Chosen for lowest maintenance under Chrome-API drift: CDP is Chrome's native protocol, so changes pass straight through the raw `cdp(method, params)` tool with zero code change; there is **no chromedriver version-matching** and **no Node driver**, so the binary freezes trivially with PyInstaller (pure-Python `websocket-client`/`pychrome`). The cost — building high-level ergonomics (waiting, click-by-selector) — is paid in the tool tickets, not borrowed from a heavy dependency.
- **Tool surface = curated high-level tools + a raw escape hatch.** Hand-picked ergonomic tools (`navigate`, `screenshot`, `evaluate_js`, …) for the common path, PLUS raw `cdp(method, params)` so the agent is never blocked when a high-level tool is missing. The escape hatch is the completeness guarantee — don't remove it to "clean up" the surface.
- **Profiles: persistent master → ephemeral per-session copy (seed-on-launch).** A single persistent *master* profile (logged in once) is **copied** into each session's fresh `user-data-dir` at launch, so every session inherits logins yet writes only to its own isolated copy. This exists because **Chrome refuses to share a `user-data-dir` across processes** (`SingletonLock`): a shared profile across parallel sessions corrupts — that was the original "multiple sessions writing the same profile" failure. Session changes are ephemeral by default; an explicit `promote` writes state back to the master **serially** (one session at a time — concurrent promote re-introduces the corruption). Seeding works because cookies are DPAPI-encrypted per *Windows user*, and all instances run as the same user, so the copied `Local State` key still decrypts. Cross-machine copies would break this — same-host only.
- **Session-state lives under `${CLAUDE_PLUGIN_DATA}`, never in the user's repo.** Per-session files `${CLAUDE_PLUGIN_DATA}/instances/<session>.json` (with an `os.tmpdir()` fallback for older Claude Code), per the `agent-serena-wrapper#22` pattern — writing runtime state into the target repo leaks it into `git status`/commits. Here the store is for **crash-recovery (reattach to a live Chrome) and orphan reaping**, not multiplexing. Open item to verify during #1: confirm `CLAUDE_PLUGIN_DATA` is actually populated for *MCP servers* (the #22 precedent was a hook) — fall back to a stable per-plugin path if not.
- **vdesktop handoff is by HWND, because vdesktop has no session concept.** `agent-vdesktop` adopts windows via `adopt_window(hwnd) → handle_id` (verified in `lib-python-vdesktop`; PID/title are fallbacks). So the wrapper's `get_instance_info` must surface the Chrome window's **HWND** (resolve it by enumerating top-level windows, matching PID + class `Chrome_WidgetWin_1`). The interop skill teaches the agent the recipe `get_instance_info → vdesktop.adopt_window(hwnd) → move/resize`. This is the *only* contract between the two plugins — there is no shared session id.
- **Chrome must run headful.** vdesktop positions a real, visible window; headless has none. Don't add a headless default.

## Contracts an agent won't infer from the tree

- **Windows-only plugin.** `OS_TARGETS = [windows]`: `release.yml`'s build matrix and `test.yml` run Windows only, and both manifests' `command` carries an explicit `.exe` (`bin/chrome-wrapper.exe`). Under Linux the plugin is expected to run against a Chrome on the host. Don't re-add the Linux matrix row unless the plugin gains a real cross-platform binary path.
- **Release is orphan-branch + marketplace dispatch.** `release.yml` (manual: Actions → release → `version=X.Y.Z`) stamps the version, builds the Windows binary, then force-pushes an orphan `release` branch holding only install-ready files and POSTs a dispatch to `Seretos/agent-marketplace`. `main` and `release` share no history — never merge between them. Clients install at the tag `agent-chrome-wrapper--vX.Y.Z`.
- **Version is pipeline-owned.** The `version` in `pyproject.toml` and both manifests is a placeholder; the workflow input is the source of truth and the stamp never lands on `main`. Don't hand-bump it.
- **Two host manifests, no `.mcp.json`.** `.claude-plugin/plugin.json` resolves its `command` via `${CLAUDE_PLUGIN_ROOT}`; `.codex-plugin/plugin.json` via `${PLUGIN_ROOT}`. Both carry an inline `mcpServers` block because neither placeholder expands in the other host. Keep the two in sync.
- **Required secret:** `MARKETPLACE_DISPATCH_TOKEN` — fine-grained PAT, `Contents: RW` + `Pull requests: RW` on `Seretos/agent-marketplace` only.
- **`assets/icon.png` is a release artifact, not just a repo file.** The dispatch payload sends a `raw.githubusercontent.com/${repo}/${TAG}/assets/icon.png` URL to the marketplace, so the file must live on the orphan `release` branch at the tagged commit — `release.yml` copies `stamped/assets/` into the staging tree for exactly that reason. Ship `assets/icon.png` from day one or the marketplace listing has no image.
- **`description.md` is a release artifact, not just a repo file.** The dispatch payload sends a `raw.githubusercontent.com/${repo}/${TAG}/description.md` URL in the `description_url` field, so the file must live on the orphan `release` branch at the tagged commit — `release.yml` copies it into the staging tree alongside `assets/`. Fill in its Key Features before cutting v0.0.1.

## Gotchas (the "why" behind the code)

- **`build.ps1` runs under Windows PowerShell 5.1, PS7, and Linux `pwsh`.** It derives `$IsWindows` from `$env:OS` (5.1 lacks the auto variable) and sets no global `$ErrorActionPreference='Stop'` (PyInstaller floods stderr, which 5.1 wraps as ErrorRecords and would trip a global Stop). The smoke step gates the build on a real MCP `initialize` handshake. The script still branches on `$IsWindows` for the (unused on this plugin) Linux path — that's intentional, leave it.
- **Native bindings need `collect_all(...)` in `chrome-wrapper.spec`** — PyInstaller misses their lazily-generated submodules otherwise. If you add a Chrome-driver library (Selenium, Playwright, pychrome, CDP client) that ships native bits, wire it in there.
